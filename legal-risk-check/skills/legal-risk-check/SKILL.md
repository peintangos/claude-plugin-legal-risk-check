# 法的リスク調査スキル

プロダクトが法律的に問題ないか、訴訟リスクがないかを予備的に調査し、開発者が見落としがちな法的リスクをスクリーニングする。

---

## リファレンス資料

`references/` ディレクトリに、複数のAIディープリサーチによる法務チェックガイドが格納されている。各カテゴリの調査時に**必ず該当する reference を Read して参照**し、リスク判定の根拠や具体的な法令条文・判例・費用目安をレポートに反映すること。

| ファイル | 主な参照場面 |
|---------|-------------|
| `references/deep-research-report.md` | 全カテゴリ。特に優先度付きチェックリスト、公開前の実務アクションプラン、権利帰属・OSS・プライバシー・消費者保護の具体手順 |
| `references/deep-research-report-by-gemini.md` | カテゴリ1(知的財産権)、カテゴリ2(OSSライセンス)、カテゴリ5(特商法)、カテゴリ7(競争法)。特に特許FTO、商標の称呼検索、景表法の直罰規定、スマホ競争促進法、AI事業者ガイドライン |
| `references/compass_artifact_wf-*.md` | カテゴリ2(OSSライセンス)、カテゴリ3(プライバシー)。特にOSSライセンスの SaaS 視点での義務整理、AGPL のネットワーク条項、商標登録の費用目安、特許法103条の過失推定 |

**参照のルール:**
- 各カテゴリの調査開始時に、上表の該当ファイルを Read する
- reference に記載された法令条文・判例（例: 著作権法27条、特許法103条）をレポートの「関連法令」欄に具体的に引用する
- reference に費用目安がある場合（例: 商標出願の印紙代、弁護士費用）は「推奨アクション」に含める
- reference の情報は2024-2026年時点のもの。法改正の可能性がある場合は「最新情報を確認」と付記する

---

## 免責事項（実行開始時に必ず表示）

スキル実行時、最初に以下を **そのまま** 出力すること:

```
⚠️ **免責事項**
この調査はAIによる予備的なスクリーニングであり、法的助言ではありません。
検出されたリスクの最終判断は、必ず弁護士等の法律専門家にご相談ください。
リスクが検出されなかった場合でも、法的問題がないことを保証するものではありません。
```

## The Job

1. 免責事項を表示する
2. プロジェクト情報を自動収集する
3. ユーザーの依頼スコープを判定する
4. フルスキャンの場合、ユーザーに5つの質問をする（選択肢形式）
5. 回答をもとにプロジェクトを自動スキャンする（**コードの実態を最優先**）
6. リスクを評価し、レポートを生成する
7. `LEGAL-RISK-REPORT.md` としてプロジェクトルートに保存する

**重要:** 法的助言は行わない。あくまで「チェックポイントの洗い出し」と「リスクの可視化」が目的。

## Step 1: プロジェクト情報の自動収集

以下のファイル・情報をプロジェクトから自動的に収集する。見つからない項目はスキップしてよい。

| 対象 | 手法 | 目的 |
|------|------|------|
| `package.json` / `requirements.txt` / `Gemfile` 等 | Read | 依存パッケージとライセンス把握 |
| `LICENSE` / `LICENSE.md` | Read | プロジェクト自体のライセンス |
| PRD / `README.md` / `CLAUDE.md` | Read | プロダクト概要の把握 |
| `.env*` ファイル | Glob で存在確認のみ（**中身は絶対に読まない**） | 外部サービス連携の有無 |
| DBスキーマ (`schema.prisma`, `migrations/`, `schema.sql` 等) | Read | 個人情報カラムの検出 |
| 認証・Cookie・トラッキング関連コード | Grep (`cookie`, `session`, `auth`, `tracking`, `analytics`, `ga(`, `gtag`) | プライバシーリスク検出 |
| API呼び出し先ドメイン | Grep (`https://`, `http://`, `fetch(`, `axios`) | サードパーティ依存の検出 |
| 法的ページ | Grep / Glob (`privacy`, `terms`, `tokushoho`, `特定商取引`, `特商法`) | 法的ページの存在確認 |
| スクレイピング・コンテンツ取得コード | Grep (`cheerio`, `puppeteer`, `playwright`, `scrape`, `crawl`, `BeautifulSoup`, `scrapy`, `rss-parser`, `rss`, `feed`) | コンテンツ収集方法の検出 |

## Step 2: スコープ判定

ユーザーの依頼内容を確認し、以下のどちらかを判定する:

### A. フルスキャン（デフォルト）
「法的リスクを調査して」「法的に問題ないか確認したい」等の包括的な依頼 → 全7カテゴリを調査。Step 3 に進む。

### B. 特定カテゴリのみ
「OSSライセンスだけチェックして」「プライバシーリスクを確認して」等の特定カテゴリに限定した依頼 → **該当カテゴリのみを深く調査する**。

特定カテゴリモードの場合:
- 質問はスキップしてよい（必要な情報はコードから自動収集）
- 該当カテゴリの調査をフルスキャンより深く行う（例: OSSなら推移的依存を含む全件調査）
- レポートはそのカテゴリに特化した形式でよい（7カテゴリテンプレートに従わなくてよい）
- ただし免責事項は必ず含める

---

## Step 3: ユーザーへの質問（フルスキャンの場合）

自動収集後、以下の5問を **一度に** 提示する。ユーザーは「1A, 2C, 3B, 4A, 5A」のように回答できる。

質問を提示する際、**自動収集で検出した事実と矛盾する可能性がある選択肢には注意を促す**こと。例えば、RSSフィード取得コードが検出されているのに質問3で「A. すべて自作コンテンツ」が選ばれる可能性がある場合、事前に「※コードからRSSフィード取得が検出されています」と補足する。

```
プロジェクトについていくつか確認させてください。

1. プロダクトの種類は？
   A. 個人の趣味・学習プロジェクト（公開予定なし）
   B. 個人の趣味・学習プロジェクト（公開予定あり）
   C. 個人開発で収益化予定あり
   D. 商用プロダクト（企業・チーム開発）
   E. OSS（オープンソースプロジェクト）

2. 個人情報の取り扱いは？
   A. 個人情報は一切扱わない
   B. メールアドレスのみ
   C. 氏名・住所等の個人情報あり
   D. 決済情報（クレジットカード等）あり
   E. 機微情報（健康・信条等）あり

3. コンテンツの収集方法は？
   A. すべて自作コンテンツ
   B. 公式APIを利用して取得
   C. Webスクレイピングで収集
   D. ユーザー生成コンテンツ（UGC）
   E. AI生成コンテンツを利用
   F. 複数該当（該当するものをカンマ区切りで）

4. 収益モデルは？
   A. 収益化なし
   B. 広告表示
   C. サブスクリプション課金
   D. 物販・EC
   E. アフィリエイト
   F. 複数該当（該当するものをカンマ区切りで）

5. 対象地域は？
   A. 日本国内のみ
   B. 日本 + アジア圏
   C. グローバル（EU含む）
   D. 未定・不明
```

## Step 4: コードの実態とユーザー回答の照合

**これはスキルの最も重要なステップ。** ユーザーの回答はあくまで自己申告であり、コードの実態が異なる場合がある。

回答を受け取ったら、Step 1 で収集した情報と照合し、矛盾がないか確認する:

- **質問3の回答が「A. すべて自作コンテンツ」だが、コードにRSSフェッチ・スクレイピング・外部API呼び出しがある** → ユーザーにソースコードは自作だがサイト上のコンテンツは外部由来であることを指摘し、**コードの実態に基づいてリスク評価を行う**
- **質問2の回答が「A. 個人情報なし」だが、DBスキーマにemail等のカラムがある** → 矛盾を指摘し、コードの実態を優先
- **質問4の回答が「A. 収益化なし」だが、広告・課金のコードがある** → 矛盾を指摘し、コードの実態を優先

矛盾が検出された場合:
1. レポート冒頭の「プロジェクト概要」セクションに矛盾を明記する
2. **コードの実態を根拠としてリスク評価を行う**（ユーザー回答でリスクを軽減しない）
3. 矛盾している項目のリスクレベルは、コードの実態に基づく適切なレベルを維持する

## Step 5: プロジェクトの自動スキャン

ユーザーの回答と **コードの実態** をもとに、以下の7カテゴリについて調査を行う。

### カテゴリ1: 知的財産権

- プロダクト名の商標リスク（WebSearchで類似サービス・商標を調査）
- UIデザインの模倣リスク（類似サービスとの比較が必要な場合は指摘）
- ビジュアルアセット（画像・フォント・アイコン）のライセンス確認
- AI生成コンテンツの著作権帰属

### カテゴリ2: OSSライセンス

**直接依存と推移的依存の両方を調査する。**

1. まず `package.json` 等の直接依存パッケージを全件チェック
2. **Node.js プロジェクトの場合は `npx license-checker --summary` を実行して推移的依存を網羅的に調査する**（Python なら `pip-licenses`、Ruby なら `license_finder` 等）
3. GPL / AGPL / SSPL 等のコピーレフト系ライセンスを検出
4. コピーレフト系が見つかった場合、依存チェーンを辿り影響範囲を特定
5. プロジェクト自体のライセンスとの互換性チェック
6. ライセンス不明パッケージの検出
7. デュアルライセンスの商用利用条件

### カテゴリ3: プライバシー・個人情報保護

- DBスキーマ内の個人情報カラム検出（email, name, address, phone, birth 等）
- Cookie・セッション管理の実装確認
- アナリティクス・トラッキングコードの検出（Google Analytics, Mixpanel 等）
- 外部送信規律（電気通信事業法2023年改正）への対応
- 質問2の回答に応じて:
  - B以上: プライバシーポリシーの存在と内容チェック
  - C以上: 個人情報保護法への準拠チェック
  - D: PCI DSS準拠の必要性を指摘
  - E: 機微情報の取扱いに関する追加規制
- 質問5の回答がC（グローバル）: GDPR対応チェック（同意管理、データ削除権等）

### カテゴリ4: 利用規約・API ToS

- 検出されたサードパーティAPI/サービスについて:
  - WebSearchでToS（利用規約）を確認
  - 商用利用の可否
  - データの再配布・二次利用制限
  - レート制限・利用量制限
  - 帰属表示の要件

### カテゴリ5: 特商法・電子商取引

質問4の回答がA以外の場合に調査:
- 特定商取引法に基づく表記の有無と内容
- 価格表示（税込表示義務）
- 返品・キャンセルポリシー
- サブスクリプションの解約方法の明示
- 広告表示の場合: 景品表示法への準拠

### カテゴリ6: コンテンツ規制

**コードの実態に基づいて調査する**（ユーザーの質問3回答だけに依存しない）:
- RSS/API経由のコンテンツ取得: 各ソースの利用規約確認、著作権法上の複製権・翻案権への該当性
- スクレイピング: robots.txt遵守、利用規約違反、著作権侵害リスク
- UGC: プロバイダ責任制限法、違法コンテンツ対応、通報機能の必要性
- AI生成: AI生成コンテンツの著作権、表示義務、AI事業者ガイドライン
- 記事本文のDB全文保存がある場合: 複製権侵害リスクとして指摘

### カテゴリ7: 競争法

質問4の回答がA以外の場合に調査:
- 景品表示法（優良誤認・有利誤認）
- 比較広告の適法性
- アフィリエイト広告のステマ規制（2023年10月施行）
- 不当表示リスク

## Step 6: リスク評価

検出された各項目に以下の5段階リスクレベルを付与する。

**判定の原則:** 迷ったら高い方のレベルを採用する。誤って低く評価してユーザーが対応を見送るリスクの方が、高く評価して不要な確認をするリスクより大きい。

| レベル | ラベル | 意味 | 判定基準 |
|--------|--------|------|----------|
| 🔴 CRITICAL | 緊急対応必要 | 明確な法令違反・重大なライセンス違反 | コードが現時点で法令に違反している、または利用規約に明確に違反している場合。例: GPL違反でプロプライエタリ配布、商用利用禁止のRSSフィードを商用再配布、個人情報を無断で第三者送信 |
| 🟠 HIGH | 高リスク | 訴訟・行政処分のリスクあり | 法令違反の可能性が高いが解釈の余地がある場合、または利用規約違反の可能性が高い場合。例: 特商法表記なしで課金、プライバシーポリシーなしで個人情報収集、記事全文のDB保存・AI要約の翻案権侵害リスク |
| 🟡 MEDIUM | 中リスク | グレーゾーン・条件付きリスク | 法的に灰色で確認が必要な場合、または将来の状況変化でリスクが顕在化する場合。例: API ToSの商用利用条件が曖昧、Cookie同意バナーなし |
| 🔵 LOW | 低リスク | ベストプラクティスからの逸脱 | 法的問題はないが改善が望ましい場合。例: ライセンス表記の不備、帰属表示の欠落 |
| ⚪ INFO | 情報提供 | 問題なし / FYI | 確認済みで問題がない場合、または参考情報 |

## スキャン時の注意事項

- `.env` ファイルの **中身は絶対に読まない**。存在確認のみ行う
- WebSearch は公開情報（ライセンス、ToS、法令）の確認にのみ使用する
- 各カテゴリの調査は **並列で実行可能なものはAgent等を使って並列に** 行い、効率的に進める
- ユーザーの回答で該当しないカテゴリは簡潔にスキップする（「質問4でAを選択のため、特商法チェックはスキップ」等）
- **コードの実態がユーザー回答と矛盾する場合は、コードの実態を優先してリスク評価を行う** — ユーザーは自分のコードの法的側面を正確に把握していない場合がある。コードが外部コンテンツを取得・蓄積・加工しているなら、ユーザーが「自作コンテンツ」と回答してもその事実に基づいて評価する
- 調査結果が不確実な場合は、リスクレベルを高めに設定し、「要確認」と明記する
- WebSearch が利用できない場合は、その旨を記載し、手動確認を推奨する

## Checklist

レポート保存前に確認:

- [ ] 免責事項を実行開始時に表示した
- [ ] プロジェクト情報を自動収集した
